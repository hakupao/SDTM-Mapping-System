## 1. はじめに

### 1.1 背景
ENSEMBLE研究では、入力整形工程から標準マッピング工程に至る変換の中で、
設定ファイルのみでは表現できない研究固有の業務ロジックが存在する。
この不足部分を研究固有関数で補完し、後続のSDTM変換品質を担保する。

### 1.2 目的
本書は以下を要求として定義する。
1. 任意条件によるデータ抽出 (`filter_df_by_field`)。
2. DM用の基本情報統合 (`DM`)。
3. 前後工程連携における研究固有ロジックの再現性・追跡性確保。

### 1.3 適用範囲
- 対象機能: 研究固有関数 `filter_df_by_field`, `DM`
- 対象連携点: 上位工程からの研究固有関数呼び出し
- 対象データ: RGST, LSVDAT, OC

---

## 2. 業務要件

### 2.1 業務フロー
1. 入力CSVを読み込む。
2. 研究固有ルールに従い抽出・統合する。
3. 生成結果を後続マッピング工程で利用可能な形式で返却する。

### 2.2 解決課題
- 複数データソースに情報が分散し、手動統合では品質が不安定。
- 研究終了日 (RFENDAT) と死亡フラグ (DTHFLG) の判定を統一ルール化する必要がある。
- Group A/B相互検証のため、要件-設計-実装-試験を追跡可能にする必要がある。

---

## 3. 機能要件

### 3.1 汎用データ抽出機能

| 要件ID | 機能名称 | 概要 | 優先度 |
|-------|----------|------|-------|
| FR-01 | 条件指定抽出 | 指定データセットに対し、項目と値の条件で抽出できること。 | 高 |
| FR-02 | 不要項目除外 | 抽出結果で全行空白の項目を出力から除外すること。 | 中 |
| FR-03 | データ型統一 | 出力データは全て文字列として扱うこと。 | 高 |

### 3.2 DM生成機能

| 要件ID | 機能名称 | 概要 | 優先度 |
|-------|----------|------|-------|
| FR-04 | データ統合 | RGST/LSVDAT/OC を `SUBJID` で統合できること。 | 高 |
| FR-05 | RFENDAT判定 | `DTHDAT` 優先、次点 `LSVDAT` で研究終了日を決定すること。 | 高 |
| FR-06 | DTHFLG判定 | `DTHDAT` が存在する場合に `Y`、それ以外は空白とすること。 | 高 |
| FR-07 | 整合性通知 | 両日付あり/両日付なし等の不整合を検知し通知できること。 | 中 |

### 3.3 連携・実装要件

| 要件ID | 機能名称 | 概要 | 優先度 |
|-------|----------|------|-------|
| FR-08 | 上位工程連携 | 上位工程から研究固有関数が呼び出せること。 | 高 |
| FR-09 | 追跡可能性 | FR -> BD -> DD -> Code -> Test の対応関係を文書で追跡できること。 | 高 |

## 4. データ要件

### 4.1 入力データ

| ファイル名称 | 必須 | 概要 | キー項目 |
|-----------|-----|------|--------|
| RGST.csv | ○ | 症例登録データ。ベースライン情報を含む。 | SUBJID |
| LSVDAT.csv | ○ | 最終生存確認データ。 | SUBJID |
| OC.csv | ○ | 転帰データ。死亡日を含む。 | SUBJID |

### 4.2 出力データ (F-DM)
必須列として以下を含むこと。
- 症例ID (SUBJID)
- 性別コード (SEXCD)
- 年齢 (AGE)
- 最終生存確認日 (LSVDAT)
- 死亡日 (DTHDAT)
- **研究終了日 (RFENDAT)**
- **死亡フラグ (DTHFLG)**

補足:
- RGST由来の追加列を保持する実装を許容するが、上記必須列の意味と算出ルールを優先すること。

## 5. 非機能要件

| 要件ID | 区分 | 要件 |
|-------|------|------|
| NFR-01 | 性能 | 数千件規模データに対して実務上許容できる時間で処理完了すること。 |
| NFR-02 | 品質 | RFENDAT/DTHFLG判定が定義ルールに一致すること。 |
| NFR-03 | 堅牢性 | 欠損があっても処理継続し、通知可能であること。 |
| NFR-04 | セキュリティ | 不要な個人情報を出力へ含めないこと。 |
| NFR-05 | 監査性 | 変更履歴・試験結果・承認状態を文書で確認できること。 |

## 6. 受入基準

| 受入ID | 対象要件 | 受入条件 |
|-------|---------|---------|
| AC-01 | FR-01, FR-02, FR-03 | 条件抽出・全空列削除・文字列型統一が再現される。 |
| AC-02 | FR-04 | RGST/LSVDAT/OC の左結合結果が期待通りである。 |
| AC-03 | FR-05 | RFENDAT が `DTHDAT > LSVDAT > 空` で決定される。 |
| AC-04 | FR-06 | DTHFLG が `DTHDAT有り=Y` で決定される。 |
| AC-05 | FR-07 | 不整合ケースで通知出力が確認できる。 |
| AC-06 | FR-08 | 上位工程連携上で研究固有関数が呼び出される。 |
| AC-07 | FR-09 | トレーサビリティマトリクスで全要件が追跡できる。 |

---

## 7. トレーサビリティ運用ルール

1. 要件IDは `FR-xx`, `NFR-xx`, `AC-xx` で管理する。
2. 基本設計では `F-xx`, ルール `R-xx` を使用し、要件IDを必ず併記する。
3. 詳細設計では `TC-xx` (試験) を定義し、FRと1対多で紐づける。
4. 実装変更時は、同日に以下を更新する。
   - 詳細設計書の該当節
   - トレーサビリティマトリクス
   - テスト実施記録
5. Group A/B相互レビューで、追跡切れ行が0件であることをリリース条件とする。
